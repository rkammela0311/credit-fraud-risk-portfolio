# Methodology Notes

Cross-cutting notes that apply to most of the projects in this portfolio.

## On synthetic data

Every dataset here is generated by `data/generate_synthetic_data.py`. The generators are tuned to produce realistic distributions:

- Credit loan default rate ≈ 8–12% (consistent with personal-loan books)
- Transaction fraud rate ≈ 0.5% (mixed CP + CNP traffic)
- Application fraud rate ≈ 5% (consistent with sub-prime origination)
- Class boundaries driven by latent risk scores with realistic feature interactions (small-business loans default more, online card-not-present is more fraud-prone, thin-file applicants are more synthetic-identity prone)

Synthetic data is never as messy as real data. Real lending data has missing values, conflicting bureau reports, regulatory misclassifications, and seasonality. The code patterns here transfer; the relative model performance probably doesn't.

## Train / validation / test discipline

Every model in this portfolio uses **time-based splits**, never just random `train_test_split` on the whole dataset. The reason is simple: in production the model is always scoring the future against patterns learned from the past. A random split lets the model peek at "future" examples in training and gives over-optimistic metrics.

Standard split:
- **Train** — 80% of in-time data
- **Validation** — 20% of in-time data, used for threshold tuning and early stopping
- **OOT (out-of-time)** — held out by date, never seen during training. The honest performance estimate.

## Why I bother with calibration

Most ML tutorials stop at AUC or F1. For credit and fraud risk, calibration is at least as important as discrimination:

- **Credit risk**: PD feeds directly into ECL (`ECL = PD × LGD × EAD`). If the model predicts 5% PD when the true rate is 10%, the bank under-provisions by half. AUC says nothing about this; calibration plots and Brier score do.
- **Fraud risk**: probabilities feed into expected-loss-based decision rules ("approve if EL < $5"). Miscalibrated scores break these rules.

Tools in this repo that address calibration:
- `model_evaluation.calibration_table()` — decile predicted vs. actual
- `model_evaluation.evaluate_binary_classifier()` — reports Brier score
- For severely miscalibrated models, `sklearn.calibration.CalibratedClassifierCV` with `method="isotonic"` is the standard fix

## Class imbalance

I use three different patterns depending on the model:

| Pattern | Where | Why |
|---|---|---|
| `class_weight="balanced"` (logistic) | PD model, scorecard | Smooth, doesn't change the loss function shape much |
| `scale_pos_weight=N_neg/N_pos` (XGBoost) | All XGB models | Lets the gradient see imbalance directly without resampling |
| Threshold tuning | Fraud models | Tune the threshold on validation for a target recall, rather than fight the imbalance during training |

I do **not** use SMOTE or random oversampling here. Both work but they tend to produce overconfident probabilities (calibration suffers) and they create synthetic minority points that don't reflect real-world boundary cases.

## PSI for monitoring

PSI (Population Stability Index) is the standard credit-risk drift metric. Once a model is in production you check, every month:

1. Score PSI between current and development sample
2. Feature PSI for every input

Industry rules of thumb:
- PSI < 0.10 — model stable, do nothing
- 0.10–0.25 — drift, investigate which features are moving
- > 0.25 — material drift, redevelop

The same logic applies to fraud, but fraud models drift faster (attackers adapt) so the monitoring cadence is weekly or daily, not monthly.

## Model risk and regulator-friendly choices

A few decisions in this portfolio that lean toward model-risk-acceptable rather than purely accuracy-maximizing:

- **Logistic regression baseline always shipped alongside the GBM challenger.** Even if the GBM wins, the logit is what gets approved first while the GBM goes through model risk review.
- **Monotonicity constraints not enabled in XGBoost** here, but they're a one-line addition (`monotone_constraints=...`) and are commonly required for credit-decision models. Regulator wants higher income to never increase predicted PD; constraint enforces it.
- **WoE-transformed inputs for the scorecard.** Loses some signal vs. raw features but produces a model that's literally a points table — auditable, interpretable, and what the lending industry has trusted for decades.
- **No deep learning.** Could improve marginal accuracy on the transaction fraud problem; not worth the explainability tradeoff for most banks unless you're operating at FAANG scale.
